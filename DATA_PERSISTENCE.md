# ความต่อเนื่องของข้อมูล (Deploy / อัปเกรดระบบ)

## หลักการสำคัญ

- **อัปเว็บ (Next.js / Vercel)** = อัปเฉพาะไฟล์และโค้ดฝั่งเบราว์เซอร์ **ไม่ลบฐานข้อมูล**
- **ข้อมูลสมาชิก ออเดอร์ ร้าน หัวใจ เกม รางวัล** อยู่ใน **PostgreSQL** ตราบใดที่ `DATABASE_URL` ชี้ไปที่ **ฐานข้อมูลเดิม** ข้อมูลจะ **ต่อเนื่อง** หลัง deploy
- **รีสตาร์ท API** = โหลดโค้ดใหม่ + รัน `initDb()` เพื่อ **เพิ่มคอลัมน์/ดัชนี** ที่ขาด (migration แบบ idempotent) **ไม่ใช่** ล้างตาราง

## สิ่งที่ไม่ควรทำถ้าอยากให้ข้อมูลไม่หาย

| การกระทำ | ผลกระทบ |
|----------|---------|
| เปลี่ยน `DATABASE_URL` ไปฐานใหม่ว่างเปล่า | ข้อมูลเก่าอยู่ DB เก่า — แอปจะเหมือน “เริ่มใหม่” |
| ลบ / รีเซ็ต volume ของ PostgreSQL โดยไม่มีสำรอง | ข้อมูลหายถาวร |
| แอดมินกด **ลบเกม** | ลบเกม + รูป + กติกา + รางวัลที่ผูก `game_id` (ออกแบบให้ลบตาม) |
| แอดมิน **บันทึกกติกาใหม่** (แทนที่ชุดเดิม) | เดิมเคยลบแถว `central_game_rules` → รางวัลที่ผูก `rule_id` ถูก CASCADE — **แก้แล้วในโค้ดล่าสุด** (สำเนา + `ON DELETE SET NULL`) |

## สิ่งที่โค้ดปัจจุบันช่วยรักษาแล้ว

- **ประวัติรางวัล (`central_prize_awards`)** — เก็บสำเนาชื่อเกม / ชุด / รายละเอียดรางวัลตอนชนะ และ `rule_id` ใช้ `ON DELETE SET NULL` เพื่อไม่ให้แถวรางวัลหายเมื่อแทนที่กติกา

## แนะนำสำหรับ production

1. ใช้ **PostgreSQL แยก**จากเครื่อง build (managed DB เช่น Neon / Supabase / Railway Postgres)
2. เปิด **สำรองอัตโนมัติ (backup)** ของผู้ให้บริการ DB
3. หลัง deploy API ดู log บรรทัด `[db] PostgreSQL schema พร้อม` — ถ้า `init failed` แก้ก่อนใช้งานจริง

## ข้อมูลที่ไม่ถาวรโดยออกแบบ

- **เซสชันเปิดป้ายใน RAM** (`centralGameSession`) — รีสตาร์ท API แล้วรอบที่กำลังเล่นอาจหาย; ความคืบหน้าถาวรอยู่ที่บันทึกรางวัล/ยอดหัวใจตาม DB ไม่ใช่ session
